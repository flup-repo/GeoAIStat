## Summary
- Build a public, read-only web app centered on a cinematic-atlas 3D globe with two launch scopes: global country view and US state drill-down where source data exists.
- Use a static-first architecture: `React 19 + Vite + TypeScript` for the app, `@react-three/fiber` plus `r3f-globe`/`three-globe` for the globe scene, and plain static JSON artifacts generated by an ingestion pipeline.
- Deploy the frontend on Vercel Hobby. Run data refresh from GitHub Actions on a daily schedule that checks a repo config and only refreshes when `cadenceDays` has elapsed; default `cadenceDays = 14`.
- Do not use a database in v1. The source set is periodic, small, and read-mostly, so a DB adds operational cost without improving the first release materially.
- Start with two provider adapters only: OpenAI Signals and Anthropic Economic Index geography data. Normalize into provider-relative scores; do not present raw values as directly comparable across providers unless the metric definitions truly match.

## Technology Choices
- Recommended globe stack: `@react-three/fiber + r3f-globe/three-globe`.
  Reason: best balance of visual control and implementation speed; direct support for polygons, heatmaps, day/night, custom materials, and React composition.
- Rejected for v1:
  - `CesiumJS`: strongest geospatial engine, but heavier and better suited to terrain/3D tiles workflows than this editorial data product.
  - `deck.gl GlobeView`: not chosen because GlobeView is still experimental and has important interaction/rendering limits.
  - `MapLibre globe`: useful for map-centric work, but weaker for a hero-grade true 3D globe experience.
- Existing reference apps support this choice:
  - `Seismic3D`, `Veins of the Earth`, `Earth’s Rivers`, and `Heat Globe` are all React/Three-oriented.
  - `DecaEarth` is Cesium-based.
  - `anymap` demonstrates the MapLibre/deck.gl branch.

## Implementation Changes
### App experience
- Single public route: `/` with state encoded in query params, not multiple pages.
- Query contract:
  - `provider`: `openai | anthropic`
  - `metric`: canonical metric id
  - `period`: ISO-like key such as `2026-02` or snapshot id
  - `mode`: `world | us`
  - `selection`: ISO3 country code or US state code
- Layout:
  - Full-viewport globe canvas as the primary surface.
  - Left floating control rail for provider, metric, period, and mode.
  - Right detail sheet for selected geography, sparkline, rank, methodology, and source attribution.
  - Bottom scrubber for time/snapshot playback.
- Visual direction:
  - Dark oceanic base, warm amber/cyan accents, atmosphere halo, subtle star field, slow autorotation on idle, film-grain/light-bloom kept restrained.
  - Desktop-first, but mobile remains functional with a bottom sheet and reduced scene complexity.
  - Add reduced-motion and low-GPU fallbacks: disable autorotation/postprocessing first; if WebGL init fails, show a 2D fallback panel with the same data.
- Implemented so far:
  - full-viewport globe surface with a cinematic point-based marker scene
  - floating control rail, detail sheet, legend, and snapshot timeline
  - reduced-WebGL fallback panel
  - URL-synced provider/metric/period/mode/selection state
- Still required:
  - actual country/state boundary fills and selection hit areas
  - richer hover states, camera choreography, and mobile bottom-sheet treatment
  - postprocessing/perf tuning only after real data and polygons are in place

### Data model and processing
- Canonical normalized row type:
  - `geoType`: `country | us_state`
  - `geoId`: ISO3 or USPS state code
  - `provider`: `openai | anthropic`
  - `metricId`: internal metric id
  - `periodId`: display period key
  - `valueRaw`: provider-native numeric value
  - `valueNormalized`: 0-100 score within the same provider/metric/period
  - `rank`: integer rank within the same provider/metric/period/geography level
  - `valueType`: `index | share | percent | score | count`
  - `unit`: optional
  - `sourceUrl`, `methodologyUrl`, `publicationDate`, `fetchedAt`
  - `comparabilityGroup`: string used to prevent invalid cross-provider comparisons
- Generated artifacts:
  - `public/data/manifest.json`: datasets, periods, checksums, last refresh, available geographies
  - `public/data/world/<provider>/<metric>.json`
  - `public/data/us/<provider>/<metric>.json`
  - `public/data/stories.json`: curated hero presets for initial camera/data states
- Geo boundaries:
  - Countries keyed by ISO3.
  - US states keyed by USPS code.
  - Join once during build; ship prejoined render data to the client.
- Implemented so far:
  - canonical typed observation/manifest/story/query schemas
  - seeded OpenAI and Anthropic sample datasets normalized to 0-100 per provider/metric/period
  - ranking and checksum generation during `build:data`
  - sample manifest/story/data artifact emission into `public/data/`
- Still required:
  - replace seeded sample values with real provider adapters that fetch and map upstream data
  - add validation for missing or partial upstream snapshots
  - join and ship render-ready geography boundary data
  - add `config/data-refresh.json` and cadence-aware refresh gating

### Refresh and deployment workflow
- Repo config file:
  - `config/data-refresh.json` with `cadenceDays`, enabled providers, output version, and force-refresh override.
- GitHub Actions:
  - Run once daily at an off-hour minute to reduce scheduling delays.
  - Step 1: read config and manifest.
  - Step 2: exit early if cadence not reached and no manual dispatch.
  - Step 3: fetch providers, normalize, validate, write processed artifacts.
  - Step 4: commit only when checksums changed.
- Vercel:
  - Auto-deploy from the default branch.
  - No server runtime required for v1.
- Manual fallback:
  - `workflow_dispatch` for emergency refreshes or source-format fixes.
  - If GitHub scheduled workflows become unreliable or get auto-disabled, migrate only the refresh job to Cloudflare Workers Cron; leave the frontend on Vercel.

## Assumptions and Defaults
- v1 is public and read-only: no auth, CMS, or user accounts.
- v1 uses only official/public provider data and provider-authored research releases.
- Cross-provider comparisons are relative unless methodologies explicitly align.
- Anthropic-style irregular releases are handled by automated checks plus manual reviewed fallback, not brittle scraping-first automation.
- Database adoption is deferred until one of these becomes true: user-generated content, ad hoc querying over larger history, admin editing workflows, or artifact size/perf makes static JSON impractical.
- If a DB is needed later, rank options as:
  - `Turso` first for simple read-heavy SQL time-series
  - `Cloudflare D1` if scheduling also moves to Cloudflare
  - `Supabase` only if auth/storage/admin tooling becomes part of the product
  - `Convex` only if the app evolves into a realtime collaborative product
